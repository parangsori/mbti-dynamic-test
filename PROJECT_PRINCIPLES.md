# PROJECT PRINCIPLES

## 1. 프로젝트 개요와 현재 상태

### 서비스 목적

- 이 서비스는 짧고 자주 할 수 있는 MBTI 스냅샷 경험을 제공한다.
- 사용자는 오늘의 성향, 미묘한 변화, 공유 가능한 결과 카드를 빠르게 확인할 수 있어야 한다.
- 개발 우선순위는 "사용자 체감 품질"과 "안전한 지속 개선"을 기준으로 잡는다.

### 현재 스택

- Frontend: React 18 + Vite
- UI: Tailwind CSS + Framer Motion
- 결과 이미지 저장: `html2canvas`
- 현재 구조는 프론트엔드 중심이며, 주요 동작은 로컬 상태와 로컬 저장소를 기반으로 한다.
- 결과 기록 백업: Supabase 익명 세션 기반 선택적 동기화. Supabase 환경변수가 없거나 저장에 실패해도 로컬 기록과 테스트 완료 흐름을 막지 않는다.
- 운영 대시보드: `admin/` 별도 Vite 앱 + Vercel Serverless API + PostHog Query API
- 관리자 접근 보호: `admin.beatblue.net` + Cloudflare Access + Vercel 환경변수 기반 서버 측 검증

### 핵심 사용자 가치

- 빠르게 시작하고 바로 결과를 본다.
- 결과 문구와 공유 카드가 반복 사용에도 지루하지 않다.
- 결과 화면, 기록, 공유 이미지가 안정적으로 동작한다.
- 기존에 정상 동작하던 경험이 사소한 수정 때문에 망가지지 않는다.

### 기본 검증 명령

- 기본 빌드 검증: `npm run build`
- 정확도 회귀 검증: `npm run test:accuracy`
- 모바일/외부기기 확인이 필요할 때: [MOBILE_TESTING.md](./MOBILE_TESTING.md)
- 기능/버그 패치는 기본적으로 `dev` 브랜치에서 먼저 작업하고 검증한 뒤 `main`에 반영한다.

## 2. 개발 원칙

### 변경 범위 원칙

- 사용자가 요청한 변경 범위를 우선으로 수정한다.
- 요청한 문제를 고치기 위해 다른 영역에 영향이 갈 수 있으면, 그 영향을 먼저 인지하고 정상 동작 영역을 보호한다.
- "이 변경을 위해 다른 쪽도 같이 흔들리는 구조"는 가능한 한 피한다.
- 사용자가 명시적으로 예외를 요청하지 않는 한, 구현 작업은 `dev`에서 시작한다. `main`에서 먼저 진행한 예외 작업은 완료 후 반드시 `dev`에도 같은 커밋을 반영한다.

### 회귀 방지 원칙

- 기존에 정상 동작 중인 기능을 망가뜨리는 구조 변경은 금지한다.
- 반복적으로 문제가 발생하는 영역은 추측으로 덧대지 말고, 이전 정상 구조와 비교해서 원인을 좁힌다.
- 문제 해결 과정에서 새 문제가 생기면, 최신 코드만 붙들지 말고 백업과 비교해 구조 차이를 확인한다.
- 공유 카드처럼 레이아웃 안정성이 중요한 영역은 문자열 길이 추정만으로 안정성을 판단하지 않는다.
- 공유 카드 핵심 카피는 정보 손실을 피하기 위해 문구를 함부로 줄이지 않고, 실제 렌더 기준으로 레이아웃 안에 맞춘다.

### 분석 우선 원칙

- 문제를 지적받으면 바로 구현부터 하지 말고, 먼저 원인과 영향 범위를 명확히 파악한다.
- 특히 레이아웃/공유 이미지/결과 문구처럼 체감형 이슈는 "왜 그렇게 보이는지"를 먼저 설명할 수 있어야 한다.

### 운영 대시보드 및 보안 원칙

- 관리자 기능은 메인 사용자 앱에 숨은 라우트로 섞지 않고, `admin/` 별도 앱과 별도 Vercel 프로젝트로 분리한다.
- 관리자 도메인은 `admin.beatblue.net`을 기준으로 하며, Cloudflare Access가 먼저 인증을 요구해야 한다.
- Vercel 도메인 검증 단계에서는 Cloudflare DNS를 `DNS only`로 둘 수 있지만, Access 보호 적용 후에는 `admin` CNAME을 `Proxied`로 켜야 한다.
- PostHog Personal API Key, Cloudflare Access Audience tag, JWKS URL, 관리자 토큰 등은 절대 프론트 번들 또는 `VITE_` 환경변수에 두지 않는다.
- PostHog Personal API Key는 Vercel admin 프로젝트의 서버 환경변수 `POSTHOG_PERSONAL_API_KEY`로만 관리하고, 최소 권한(`query:read` 중심)으로 생성한다.
- 메인 사용자 서비스의 PostHog 브라우저 SDK에는 `phc_` Project token만 `VITE_POSTHOG_KEY`로 넣는다. `phx_` Personal API Key를 `VITE_` 변수에 넣으면 프론트 번들에 노출되므로 즉시 폐기/재발급한다.
- 메인 사용자 서비스의 PostHog 수집 호스트는 reverse proxy `VITE_POSTHOG_HOST=https://e.beatblue.net`을 사용한다.
- Supabase 클라이언트에는 publishable/anon key만 `VITE_SUPABASE_*` 변수로 둔다. service role key, secret key, DB password는 절대 `VITE_` 변수나 프론트 번들에 두지 않는다.
- Supabase 결과 저장은 로컬 기록의 보조 백업으로만 시작한다. 서버 저장 성공 여부 때문에 테스트 완료, 결과 표시, 공유 카드 저장 흐름이 막히면 안 된다.
- 클라이언트는 임의 HogQL/query를 서버로 보낼 수 없어야 하며, 서버리스 API는 코드에 고정된 집계 쿼리만 실행한다.
- 운영 대시보드는 집계 숫자만 표시한다. 사용자별 이벤트 원문, distinct id, 이름, 생년월일, 원본 프로필 데이터는 응답하거나 화면에 노출하지 않는다.
- `ADMIN_DASHBOARD_TOKEN`은 로컬 개발 또는 비상 fallback용이다. production의 주 인증 수단으로 사용하지 않는다.
- Cloudflare Access 설정이 누락되었을 때 admin API는 열리지 않고 `admin_auth_not_configured`처럼 닫힌 상태로 실패해야 한다.
- admin 배포 변경 후에는 비로그인 상태에서 `https://admin.beatblue.net`과 `/api/admin/metrics`가 Cloudflare Access 로그인으로 리다이렉트되는지 확인한다.
- admin 환경변수나 보안 설정을 바꾼 뒤에는 Vercel admin 프로젝트를 반드시 redeploy해야 한다.

## 3. 테스트 원칙

### 기본 원칙

- 테스트는 중요하지만, 시간과 크레딧을 과하게 소모하는 비효율적인 자동화는 지양한다.
- 기본은 **짧은 자동 검증 우선**으로 간다.
- 추천 기본 조합:
  - `npm run build`
  - `npm run test:accuracy`

### 사용자 빠른 확인으로 전환해야 하는 경우

- 공유 이미지 저장 결과
- 레이아웃 체감 이슈
- 긴 문구/특정 문구에서만 드러나는 화면 문제
- 자동화 검증 세팅 자체가 실제 수정보다 오래 걸리는 경우

이 경우에는:

- 자동화에 과도한 시간을 쓰지 않는다.
- 먼저 사용자에게 "이 테스트는 길어질 수 있다"는 점을 알린다.
- 사용자가 빠르게 확인하는 쪽이 더 효율적이면 그 방식으로 전환한다.

### 시간 공지 원칙

- 테스트가 1분 이상 걸릴 가능성이 있거나 여러 단계 자동 확인이 필요하면 먼저 알린다.
- 긴 검증을 무작정 시작하지 않는다.

### 렌더 기반 UI 검증 원칙

- 긴 문구, 공유 카드, 저장 이미지처럼 실제 브라우저 렌더링에 따라 결과가 달라지는 문제는 문자 수 추정만으로 검토하지 않는다.
- 이런 문제는 전체 E2E 대신, 해당 문제 지점만 겨냥한 소형 브라우저 렌더 검증을 우선 사용한다.
- 공유 카드 핵심 hook은 실제 렌더 기준으로 2줄 이하를 보장해야 하며, 아래 레이아웃을 밀지 않아야 한다.

### 장기 테스트 구조 기준

- 상시 유지해야 할 테스트 축:
  - 빌드 성공 여부
  - MBTI 정확도 회귀 여부
  - 공유 카드/레이아웃 핵심 시나리오 체크리스트
- admin 운영 대시보드 작업 시 추가 확인:
  - `npm --prefix admin run build`
  - `node --check admin/api/admin/metrics.js`
  - `node --check admin/scripts/verify-env.mjs`
  - 빌드 산출물에서 `POSTHOG_PERSONAL_API_KEY`, `ADMIN_DASHBOARD_TOKEN`, `phx_` 같은 민감 문자열이 없는지 검색
  - 비인증 외부 접근이 Cloudflare Access 로그인으로 리다이렉트되는지 확인
- 향후 자동화는 "시간 대비 효율이 높은 것"만 추가한다.

## 4. 백업 및 복구 원칙

### 백업 기본 원칙

- 모든 repo 변경 전 작업 단위 백업을 기본으로 한다.
- 사소한 수정도 예외로 두지 않는다.
- 백업 이름 규칙은 다음을 기본으로 한다:

`backups/<작업명>-YYYYMMDD/`

예시:
- `backups/share-card-layout-20260424/`
- `backups/result-copy-refresh-20260424/`

### 백업 방식

- 관련 파일만 작업 단위로 백업한다.
- 새 파일 추가만 있는 작업이라면, 작업용 백업 폴더만 먼저 생성해도 된다.
- 큰 구조 변경 전에는 현재 정상 파일을 반드시 백업한다.

### 복구/비교 원칙

- 문제가 반복되면 현재 코드 단독 분석을 금지한다.
- 먼저 백업본과 diff 비교를 한다.
- 필요하면 백업 기준으로 구조를 되돌린 후, 필요한 수정만 최소 범위로 다시 적용한다.
- "계속 덧대는 수정"보다 "정상 구조로 복귀 후 최소 수정"을 우선한다.

### 내부 검토 리포트

- `moodtype-lab-review-report.*` 파일은 코드 품질·확장성·보안·성능에 대한 내부 검토 리포트이다.
- 이 파일은 로컬 참고용으로만 유지하며, Git에 푸시하지 않는다(`.gitignore`에 등록).
- 개선 작업 시 이 리포트의 우선순위 제안을 참고한다.

## 5. 버전 및 변경 이력 원칙

### 버전 반영 대상

- 사용자 체감 기능 변화
- 시스템적으로 의미 있는 개선
- 운영상 중요한 구조 변화
- 서비스 흐름, 공유 경험, 결과 신뢰도처럼 핵심 경험에 영향을 주는 수정

### 버전 반영 제외

- 단순 버그 제거
- 미세한 패치
- 내부 정리만 있는 변경
- 서비스 체감 변화가 거의 없는 단건 수정

### CHANGELOG 운영 기준

- 기준 위치: [src/lib/constants.js](./src/lib/constants.js)
- CHANGELOG는 "사용자가 느끼는 변화" 중심으로 기록한다.
- 한 번의 작업에서 사소한 수정이 여러 개 있더라도, 사용자 체감 변화가 없다면 버전을 올리지 않는다.
- 여러 수정이 누적되더라도 의미 있는 단위가 되었을 때 묶어서 반영할 수 있다.
- 반복적인 여백, 색상, 카드 폭, 모션 수치 조정은 개별 항목으로 늘어놓지 않고 "사용성 개선", "레이아웃 안정성 패치", "스와이프 경험 보완"처럼 묶어 기록한다.

## 6. 현재 로드맵 + 최근 이슈 로그

### 현재 우선순위

1. 결과 화면 및 공유 카드 안정성 유지
2. 반복 사용 시 결과 문구/테마/질문 신선도 개선
3. 변경 작업 시 회귀 방지와 백업-비교 체계 고정
4. PostHog 기반 모바일 운영 대시보드로 유입, 시작, 완료, 공유, 재방문 지표를 계속 확인
5. Supabase 기반 결과 기록 백업을 로컬 우선 구조로 안정화
6. 이후 백엔드/수익화 작업 전, 프론트 경험 품질과 운영 지표 판단 흐름 안정화

### 지속 주의할 문제

- 공유 카드 레이아웃은 긴 문구에서 쉽게 무너질 수 있다.
- 결과 화면은 한 곳 수정이 다른 정상 영역에 영향을 주기 쉬우므로 범위 통제가 중요하다.
- 자동 캡처/자동 UI 검증은 시간과 크레딧을 많이 소모할 수 있으므로 항상 효율성을 먼저 판단한다.
- admin 대시보드 보안은 "URL을 숨기는 것"으로 해결하지 않는다. 반드시 Cloudflare Access, 서버 전용 환경변수, 집계 응답 제한을 함께 유지한다.
- Vercel에서 `admin.beatblue.net`이 `DNS Change Recommended`로 보이더라도, Access 보호를 위해 Cloudflare 프록시가 필요한 상태인지 먼저 판단한다.
- PostHog 이벤트/대시보드 지표를 추가할 때는 개인정보 payload가 포함되지 않는지 먼저 확인한다.

### 최근 해결/미해결 이슈

- 결과 하단 궁합 카드의 긴 문구 정렬 문제를 구조적으로 완화했다.
- 같은 MBTI 반복 결과에도 문구/테마/질문 변주가 적용되도록 확장했다.
- 질문 화면에 모바일 스와이프, PC 드래그, 키보드 방향키 답변과 직전 1문항 되돌리기를 추가했다.
- 스와이프 답변은 질문 카드 자체가 아니라 좌우 답변 카드 묶음을 조작 대상으로 삼아야 사용자가 선택 대상을 더 쉽게 이해한다.
- 질문 진행 중에는 하단 version 배지를 숨기고, 답변 카드는 모바일에서 세로 전체 폭을 우선해 가독성과 조작감을 지킨다.
- 답변 카드는 모바일에서 좌우 어긋남으로 방향성을 주되, 강한 겹침보다 텍스트 가독성을 우선한다.
- 답변 카드는 좌우 방향성을 줄 때도 겹치지 않게 두고, 80% 폭과 충분한 상하 여백으로 질문/답변 구분감을 유지한다.
- 답변 카드는 동일 높이를 유지하고, 카드 묶음이 아니라 사용자가 잡은 개별 카드만 스와이프되도록 한다.
- 답변 영역은 카드만 떠 보이지 않도록 은은한 선택 영역 배경과 방향별 accent를 사용하고, 스와이프는 가볍고 즉각적인 반응을 우선한다.
