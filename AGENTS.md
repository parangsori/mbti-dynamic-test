# moodtype-lab Working Entry

이 저장소에서 새 채팅을 시작하면, 다른 분석보다 먼저 아래 순서로 진행합니다.

1. `PROJECT_PRINCIPLES.md`를 먼저 읽고 현재 원칙, 로드맵, 최근 이슈를 파악한다.
2. 이번 요청과 직접 관련된 현재 변경 파일을 확인한다.
3. 관련 `backups/` 폴더와 이전 정상 구조를 비교할 필요가 있는지 먼저 판단한다.
4. 필요한 최소 테스트만 선택하고, 테스트가 길어질 가능성이 있으면 먼저 사용자에게 알린다.
5. admin/운영 대시보드 관련 요청이면 `admin/README.md`와 `admin/DEPLOYMENT_CHECKLIST.md`를 함께 읽고, Cloudflare Access, Vercel 환경변수, PostHog API 키 노출 위험을 먼저 점검한다.
6. 기능/버그 패치는 기본적으로 `dev` 브랜치에서 먼저 작업하고 검증한 뒤 `main`에 반영한다. 사용자가 `main` 선반영을 명시한 예외 작업은 완료 후 반드시 `dev`에도 같은 커밋을 반영한다.

핵심 강제 원칙:

- 새 채팅에서는 기존 맥락을 추정하지 말고 원칙 문서부터 확인한다.
- 구현 작업은 사용자가 명시적으로 예외를 요청하지 않는 한 `dev`에서 시작한다.
- 사소한 변경 포함 모든 repo 변경 전 작업 단위 백업을 먼저 만든다.
- 요청받은 범위만 수정하고, 기존 정상 동작 영역에 회귀를 만들지 않는다.
- 문제가 반복되면 현재 코드만 보지 말고 백업본과 diff 비교를 우선한다.
- 자동 검증은 짧고 효과적인 범위로 유지하고, 시간이 길어지는 UI 확인은 사용자 빠른 테스트로 전환할 수 있다.
- 운영 대시보드와 관리자 접근 작업에서는 URL 은닉을 보안으로 간주하지 않는다. Cloudflare Access 보호, 서버 전용 환경변수, 집계 응답 제한, 비인증 접근 차단을 모두 확인한다.
- PostHog Personal API Key, Cloudflare Access Audience tag/JWKS URL, 관리자 토큰은 절대 `VITE_` 변수나 프론트 번들에 두지 않고, 채팅/문서/커밋에 실제 값을 남기지 않는다.
- `moodtype-lab-review-report.*` 파일은 로컬 참고용 내부 검토 리포트이다. Git에 푸시하지 않으며, `.gitignore`에 등록된 상태를 유지한다.

상세 운영 기준, 버전 반영 기준, 현재 우선순위는 `PROJECT_PRINCIPLES.md`를 기준으로 따른다.
